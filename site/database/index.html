    Baza de Date FOD

    Prezentare Generală

    Sistemul FOD (Front Office Digital) utilizează Entity Framework Core pentru gestionarea datelor. Baza de date este structurată pentru a suporta procesarea cererilor digitale pentru serviciile guvernamentale din Moldova.

    Arhitectura

    Context Principal

    • FodDbContext - Context-ul principal Entity Framework care gestionează toate entitățile
    • IFodDbContext - Interfața pentru dependency injection și testare

    Entitate de Bază

    Toate entitățile moștenesc din BaseEntity care oferă câmpuri comune de audit: - Id (Guid) - Identificator unic - CreateDate - Data creării - UpdateDate - Data ultimei actualizări - CreateUserId - ID utilizator care a creat înregistrarea - UpdateUserId - ID utilizator care a actualizat ultima dată - CreateUserName - Nume utilizator creator - UpdateUserName - Nume utilizator actualizator

    Categorii de Entități

    1. Entități Principale de Cereri

    • FodRequest - Cererea principală depusă de cetățean
    • FodServiceRequest - Cereri individuale de servicii în cadrul unei cereri principale
    • FodRequestType - Tipuri de cereri disponibile
    • FodAdditionalRequest - Cereri adiționale

    2. Entități de Servicii

    • FodService - Serviciile disponibile în sistem
    • FodServiceProvider - Furnizorii de servicii
    • FodServiceDocumentType - Tipuri de documente necesare per serviciu

    3. Entități de Utilizatori și Profile

    • UserProfile - Profiluri de utilizatori
    • FodUserProfileOrganization - Asocieri utilizator-organizație
    • FodUserProfileRole - Roluri utilizatori

    4. Entități de Documente

    • File - Metadate fișiere
    • FileContent - Conținut fișiere
    • SignableDocument - Documente ce pot fi semnate
    • FodResponseDocument - Documente răspuns

    5. Entități de Plăți

    • FodRequestPayment - Plăți pentru cereri
    • FodAdditionalInvoice - Facturi adiționale
    • ServiceRequestDiscount - Reduceri aplicate

    6. Entități de Livrare

    • ServiceRequestDelivery - Detalii livrare
    • ServiceRequestDeliveryStatusLog - Istoric status livrare
    • FodPickupLocation - Locații de ridicare

    7. Entități de Semnături

    • FodSignature - Semnături digitale
    • FodUserSignature - Semnături utilizatori
    • FodRequestJsonSignature - Semnături JSON

    8. Entități de Date Adiționale

    • FodAdditionalData - Date adiționale generice
    • FodAdditionalDataTypeValue - Valori pentru tipuri de date
    • JsonSchema - Scheme JSON pentru validare

    9. Entități Specializate

    • FodApostilationDetails - Detalii apostilare
    • FodNotificationType - Tipuri de notificări
    • WorkingProgram - Program de lucru
    • ExceptionWorkingProgram - Excepții program lucru

    Diagrama Relațiilor Principale

    FodRequest (Cerere Principală)
    │
    ├── FodServiceRequest (Cereri de Servicii)
    │   ├── FodService (Serviciu)
    │   ├── FodServiceProvider (Furnizor)
    │   └── FodResponseDocument (Documente Răspuns)
    │
    ├── FodRequestPayment (Plăți)
    │   └── FodAdditionalInvoice (Facturi)
    │
    ├── ServiceRequestDelivery (Livrare)
    │   └── ServiceRequestDeliveryStatusLog (Istoric)
    │
    └── FodRequestFile (Fișiere Atașate)
        └── File (Metadate)
            └── FileContent (Conținut)

    Enumerări Importante

    Status-uri Cereri

    • FodRequestStatus - Status-uri pentru cererea principală
    • FodServiceRequestStatus - Status-uri pentru cereri de servicii
    • PaymentStatus - Status-uri plăți
    • DeliveryStatus - Status-uri livrare

    Tipuri

    • PersonType - Tip persoană (fizică/juridică)
    • OnBehalfOnEnum - În numele cui se depune cererea
    • AdditionalDataType - Tipuri de date adiționale
    • DocumentTypeEnum - Tipuri de documente

    Convenții și Bune Practici

    1. Naming Convention
    2. Entitățile încep cu prefix Fod pentru entități specifice FOD
    3. Proprietățile folosesc PascalCase

    4. Foreign Key-urile se termină cu Id

    5. Relații

    6. Folosește atribute [ForeignKey] pentru relații explicite
    7. Colecțiile folosesc IList sau ICollection

    8. Navigare bidirecțională unde este necesar

    9. Audit Trail

    10. Toate entitățile moștenesc din BaseEntity

    11. Câmpurile de audit sunt populate automat

    12. Validare

    13. Validarea se face la nivel de aplicație
    14. Constrângerile de bază sunt definite în entități

    Migrații

    Migrațiile Entity Framework nu sunt stocate în repository. Pentru actualizări de schemă: 1. Generează migrări local folosind dotnet ef migrations add 2. Aplică migrările în mediile de deployment 3. Documentează schimbările majore de schemă

    Performanță

    Pentru optimizare: - Folosește Include() pentru eager loading - Implementează paginare pentru liste mari - Folosește proiecții pentru query-uri read-only - Indexează câmpurile folosite frecvent în filtrări

    Securitate

    • Nu expune direct entitățile în API-uri
    • Folosește DTO-uri pentru transfer de date
    • Implementează row-level security unde este necesar
    • Auditează toate operațiile CRUD
